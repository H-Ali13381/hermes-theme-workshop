### Known code bugs found during review

1. materialize_kvantum fallback (line ~1368):
   ```python
   kvantum_theme = "kvantum-dark"   # BUG: not a valid Qt6 style name
   ```
   The skill and ricer-kde SKILL.md both document this as wrong. The fallback
   itself is the documented bug (widgetStyle=kvantum-dark silently falls to Breeze).
   No test currently guards this regression.

2. materialize_kde_lockscreen — kreadconfig break-before-check (line ~1571):
   ```python
   for tool in ["kreadconfig6", "kreadconfig5"]:
       if cmd_exists(tool):
           rc, out, _ = run_cmd(...)
           if rc == 0 and out:
               prev_theme = out
           break   # <-- break is OUTSIDE the inner if, but INSIDE the cmd_exists block
   ```
   This breaks out of the loop after the first tool attempt regardless of whether
   it returned a result. If kreadconfig6 is installed but returns empty (e.g. key
   not set yet), kreadconfig5 is never tried. The existing tests mock cmd_exists=True
   uniformly so this is masked. Low risk in practice (kreadconfig6 is always present
   on Plasma 6) but the pattern should be consistent with how snapshot_kde_state
   does it (checks `if scheme: break`).

3. icon_theme field in design schema (line ~65, DEFAULT_DESIGN_SYSTEM):
   The schema defines `icon_theme` as a required KDE field. discover_apps() does NOT
   register an `icon_theme` key, and APP_MATERIALIZERS has no `icon_theme` entry.
   The field is read but never applied. This is a silent no-op — users setting
   icon_theme in their design JSON get nothing. Should either be implemented or
   explicitly documented as a known gap.

4. ricer-kde SKILL.md discover_apps block is stale:
   The SKILL.md shows only kvantum/plasma_theme/cursor in the "always ensure" block,
   but the real code (line 159) also registers kde_lockscreen. The skill doc is missing
   this entry and would cause confusion when adding new sub-systems.

## Validation Plan — One Test File Per KDE Element

Each section below is a self-contained test plan. Tests use unittest.mock.patch
to isolate from the live KDE environment. All tests should be placed in:
  tests/test_kde_materializers.py

### 1. materialize_kde (colorscheme)

Goal: verify .colors file is generated with decimal RGB (not hex), colorscheme
is applied via plasma-apply-colorscheme, and backup is created.

Tests to write:
  a) test_kde_colorscheme_dry_run_no_writes
     - patch HOME, BACKUP_DIR, cmd_exists=True, run_cmd returns (0,"PrevScheme","")
     - call materialize_kde(design, backup_ts, dry_run=True)
     - assert exactly 1 change with action="dry-run" and "previous_colorscheme" set
     - assert no files written (colorscheme_path does not exist)

  b) test_kde_colorscheme_writes_decimal_rgb
     - patch HOME to tmpdir, mock kwriteconfig6, plasma-apply-colorscheme
     - call materialize_kde(design, backup_ts)
     - read generated .colors file
     - assert NO hex (#rrggbb) values present
     - assert decimal comma-separated RGB tuples present (regex: r"\d+,\d+,\d+")

  c) test_kde_colorscheme_applies_via_plasma_apply_colorscheme
     - capture run_cmd calls
     - assert "plasma-apply-colorscheme" appears in call args
     - assert arg is the scheme NAME (not a file path)

  d) test_kde_colorscheme_bounces_to_breezeClassic_when_scheme_unchanged
     - set snapshot_kde_state to return same name as new scheme
     - assert run_cmd called with "BreezeClassic" before re-applying the new scheme

  e) test_kde_colorscheme_backup_created_before_write
     - assert kdeglobals_backup path exists in changes and points to backup dir
     - assert backup was created before the new .colors was written (ordering check
       via call order on run_cmd side_effect list)

  f) test_kde_colorscheme_no_hex_in_generated_file (parametrize over all presets)
     - for each PRESET in PRESETS: generate the .colors content and assert no #rrggbb

### 2. materialize_kvantum

Goal: verify kvantum.kvconfig is written with correct theme name, widgetStyle is
set to exactly "kvantum" (never "kvantum-dark"), and KWin is reconfigured.

Tests to write:
  a) test_kvantum_dry_run_returns_change_with_theme_name
     - dry_run=True
     - assert change["theme"] == design["kvantum_theme"]
     - assert no files written

  b) test_kvantum_config_written_with_correct_format
     - real write to tmpdir
     - assert kvantum.kvconfig contains "[General]\ntheme=<name>"

  c) test_widget_style_is_kvantum_not_kvantum_dark
     - CRITICAL regression guard
     - capture kwriteconfig6 call args
     - assert "kvantum" in args and "kvantum-dark" NOT in args
     - this guards the known bug at line ~1410

  d) test_kvantum_fallback_does_not_write_kvantum_dark
     - design has no "kvantum_theme" key
     - assert materialize_kvantum returns empty list (no-op) OR
       assert it does NOT write "kvantum-dark" to either file
     - NOTE: current code writes kvantum_theme="kvantum-dark" as the theme name in
       kvantum.kvconfig. The real bug is widgetStyle, which is separately set to "kvantum".
       Test should verify the widgetStyle path is always "kvantum" regardless of kvantum_theme value.

  e) test_kvantum_qdbus_reconfigure_called_after_write
     - assert qdbus6 call with "org.kde.KWin /KWin reconfigure" appears in run_cmd calls
     - assert it is called AFTER the kvantum.kvconfig write (not before)

  f) test_kvantum_backup_of_kvconfig_not_kdeglobals
     - assert change record has "backup" pointing to kvantum/ subdir (not kde/)
     - assert kdeglobals is NOT backed up by kvantum materializer (stale backup bug)

### 3. snapshot_kde_state

Goal: verify all 7 fields are captured, fallback chains work, and missing keys
don't crash.

Tests to write:
  a) test_snapshot_returns_all_seven_fields
     - mock all kreadconfig6 calls with known values
     - assert dict has keys: active_colorscheme, look_and_feel, kvantum_theme,
       widget_style, plasma_theme, cursor_theme, wallpaper, wallpaper_plugin

  b) test_snapshot_falls_back_to_lookandfeel_when_colorscheme_missing
     - kreadconfig6 returns empty for ColorScheme but value for LookAndFeelPackage
     - assert active_colorscheme is None, look_and_feel has the value

  c) test_snapshot_reads_wallpaper_plugin_from_appletsrc
     - write tmpdir appletsrc with both Image= and Wallpaperplugin= lines
     - assert wallpaper_plugin matches

  d) test_snapshot_strips_file_uri_from_wallpaper_path
     - appletsrc has Image=file:///home/user/wall.jpg
     - NOTE: snapshot_kde_state does NOT strip file:// — that's _snapshot_current_wallpaper.
       This test should confirm the raw value is preserved (and document the stripping
       happens downstream). Or add stripping here for consistency.

  e) test_snapshot_returns_none_for_missing_kvantum_config
     - no kvantum.kvconfig file in tmpdir
     - assert kvantum_theme is None (no exception)

### 4. materialize_cursor

Goal: verify cursor theme is set via both kwriteconfig6 and plasma-apply-cursortheme,
backup is created, and undo restores previous value.

Tests to write:
  a) test_cursor_dry_run_no_writes
  b) test_cursor_writes_kcminputrc_with_theme_name
  c) test_cursor_calls_plasma_apply_cursortheme
  d) test_cursor_skips_gracefully_when_no_cursor_theme_in_design
     - design has no "cursor_theme" key
     - assert empty list returned without error
  e) test_cursor_previous_value_captured_in_change_record
     - previous_cursor should be set from kreadconfig6 result

### 5. materialize_plasma_theme

Goal: verify plasmarc is written, plasma-apply-desktoptheme is called, and
skips cleanly when plasma_theme not in design.

Tests to write:
  a) test_plasma_theme_dry_run_no_writes
  b) test_plasma_theme_writes_plasmarc_group_theme_key_name
     - assert kwriteconfig6 called with --group Theme --key name <value>
     - NOT --group General (wrong group used in some versions)
  c) test_plasma_theme_calls_plasma_apply_desktoptheme
  d) test_plasma_theme_skips_when_not_in_design
     - design = {"palette": {...}} with no plasma_theme key
     - assert empty list, no files touched
  e) test_plasma_theme_backup_is_plasmarc_not_kdeglobals

### 6. materialize_konsole (new)

No tests currently exist. Konsole has its own .colorscheme format and profile
activation. Tests to write:

  a) test_konsole_writes_colorscheme_file_in_local_share
     - assert file written to ~/.local/share/konsole/hermes-<name>.colorscheme
  b) test_konsole_colorscheme_format_has_required_sections
     - assert [Background], [Foreground], [Color0] through [Color15] sections present
  c) test_konsole_dry_run_no_writes
  d) test_konsole_default_profile_updated_in_konsolerc
  e) test_konsole_previous_profile_captured_for_undo_hint

### 7. undo() — KDE restore paths

Goal: verify undo correctly restores KDE-specific state from a manifest.

Tests to write:
  a) test_undo_re_applies_previous_colorscheme
     - manifest with kde change, previous_colorscheme="BreezeClassic"
     - assert plasma-apply-colorscheme called with "BreezeClassic"

  b) test_undo_restores_widget_style_via_delete_not_empty_string
     - CRITICAL: kwriteconfig6 --delete must be used, not --key widgetStyle ""
     - manifest with kvantum change, previous_widget_style=None
     - assert "--delete" appears in kwriteconfig6 call args

  c) test_undo_restores_cursor_theme
     - manifest with cursor change, previous_cursor="Breeze"
     - assert plasma-apply-cursortheme called with "Breeze"

  d) test_undo_restores_plasma_theme
     - manifest with plasma_theme change, previous_theme="default"
     - assert plasma-apply-desktoptheme called OR kwriteconfig6 sets Theme.name

  e) test_undo_restores_lock_screen_theme
     - manifest with kde_lockscreen change, previous_theme="org.kde.breeze.desktop"
     - assert kwriteconfig6 called with that value

  f) test_undo_skips_dry_run_manifest_with_error
     - manifest has dry_run=True
     - assert status="error", no side effects

  g) test_undo_marks_manifest_as_undone_after_success
     - assert manifest written back with "undone": True

### 8. discover_apps KDE sub-system registration

Goal: verify that when KDE is detected, all 4 sub-systems are registered.

Tests to write:
  a) test_discover_apps_kde_registers_kvantum_plasma_theme_cursor_lockscreen
     - mock cmd_exists to return True for plasmashell
     - call discover_apps()
     - assert "kvantum", "plasma_theme", "cursor", "kde_lockscreen" all in result

  b) test_discover_apps_kde_subsystems_absent_without_kde
     - mock cmd_exists to return False for plasmashell/kwriteconfig6/kwriteconfig5
     - assert none of the four sub-system keys appear

## Existing Gaps to Fix (not new tests — code fixes)

### Fix 1: materialize_kvantum fallback writes "kvantum-dark" as theme
File: scripts/ricer.py ~L1366-1368
Current:
  if not kvantum_theme:
      kvantum_theme = "kvantum-dark"
Fix options:
  Option A — return empty list (no-op, safe): if not kvantum_theme: return changes
  Option B — log a warning and skip: the user hasn't configured a Kvantum theme,
             so silently installing the wrong one is worse than doing nothing.
Recommended: Option A. Document in ricer-kde SKILL.md.

snapshot_kde_state pattern: only break if a value was retrieved.
