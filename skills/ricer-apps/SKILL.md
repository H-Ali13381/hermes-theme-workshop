# ricer-apps — Application Materializers

Part of the hermes-ricer suite. Each section covers one app's theming approach,
config paths, pitfalls, and the materializer contract.

---

## Materializer Contract

Every materializer function must:

1. Accept `(design: dict, backup_ts: str, dry_run: bool = False) -> list[dict]`
2. On `dry_run=True`: return a list of change dicts describing what WOULD happen (no writes)
3. Before any write: call `backup_file(path, backup_ts, "app/filename")` for each file touched
4. Inject include/source directives with a `# hermes-ricer` marker line for clean undo
5. Return a list of change dicts, one per file written or action taken

## kitty

Config: `~/.config/kitty/`
Files written: `theme.conf` (palette), `kitty.conf` (injected include line)

### How it works

1. Writes `~/.config/kitty/theme.conf` with all color assignments
2. Injects `include theme.conf  # hermes-ricer` at the end of `kitty.conf` (once only)
3. Backs up both `theme.conf` AND `kitty.conf` before any write

### Color mapping

```
background       → background
foreground       → foreground
selection_bg     → primary (with alpha)
selection_fg     → background
cursor           → accent
url_color        → primary
color0-7         → [background, danger, success, warning, primary, secondary, accent, foreground]
color8-15        → intense variants (slightly brighter)
```

### Live reload

kitty does not require a restart — it picks up config changes automatically via `SIGUSR1`:

    kill -SIGUSR1 $(pgrep -x kitty)

ricer.py should send this signal after writing.

### Undo

Restore `theme.conf` from backup dir. Remove the injected include line from `kitty.conf`
(matched by the `# hermes-ricer` marker).

## Konsole

Config: `~/.local/share/konsole/` (profiles), `~/.config/konsolerc` (default profile)
Files written: `hermes-ricer.profile`, `hermes-ricer-colors.colorscheme`

### How it works

1. Writes a `.colorscheme` file to `~/.local/share/konsole/hermes-ricer-colors.colorscheme`
2. Writes a `.profile` file to `~/.local/share/konsole/hermes-ricer.profile` referencing the colorscheme
3. Sets the default profile in `konsolerc` via kwriteconfig6

### CRITICAL: DefaultProfile activation

The profile file alone is not enough. You MUST also write the DefaultProfile key:

    kwriteconfig6 --file konsolerc --group "Desktop Entry" --key DefaultProfile hermes-ricer.profile

Without this, Konsole ignores the new profile entirely on next launch.

### Konsole color scheme format

Uses decimal RGB (r,g,b) — same as KDE .colors files. Always use `hex_to_rgb()` to convert.

Example entry:

    [Background]
    Color=12,18,32

### Undo

Restore the previous `DefaultProfile` key in `konsolerc`. The old profile file is not
deleted — it remains available. Konsole requires a new session to activate changes;
running instances are not affected by undo.

### Pitfall: Konsole requires a new session

Terminal colors activate only in new Konsole sessions after theming. Undo records
the previous profile name but cannot force-switch a running instance.

## waybar

Config: `~/.config/waybar/`
Files written: `style-hermes.css`, injected `@import` in `style.css`

### How it works

1. Writes `~/.config/waybar/style-hermes.css` with full CSS palette
2. Injects `@import "style-hermes.css";  /* hermes-ricer */` at the top of the main config
3. Backs up `style-hermes.css` AND the main style file before writing

### CSS structure

```css
/* Generated by hermes-ricer — <theme-name> */
* {
    font-family: "<ui_font>", sans-serif;
    font-size: 14px;
}

window#waybar {
    background-color: <background>;
    color: <foreground>;
    border-bottom: 2px solid <primary>;
}

#workspaces button { ... }
#workspaces button.active { background-color: <primary>; color: <background>; }
#clock, #battery, #network { ... }
```

### Live reload

    pkill -SIGUSR2 waybar

ricer.py should send this after writing.

### Pitfall

waybar CSS `@import` injection must deduplicate — check for existing `# hermes-ricer` marker
before injecting to avoid stacking multiple import lines on repeated runs.

## rofi

Config: `~/.config/rofi/`
Files written: `hermes-theme.rasi`, injected `@import` in `config.rasi`

### How it works

1. Writes `~/.config/rofi/hermes-theme.rasi` with a complete rofi theme
2. Injects `@import "hermes-theme.rasi"  /* hermes-ricer */` into `config.rasi`

### rasi structure

```rasi
/* Generated by hermes-ricer */
* {
    background: <background>;
    foreground: <foreground>;
    primary: <primary>;
    accent: <accent>;
    surface: <surface>;
    muted: <muted>;
}

window {
    background-color: @background;
    border: 2px solid;
    border-color: @primary;
    border-radius: 8px;
}

inputbar { background-color: @surface; }
entry { background-color: transparent; text-color: @foreground; }
listview { background-color: transparent; }
element selected { background-color: @primary; text-color: @background; }
```

### Pitfall

`config.rasi` import injection: check for existing `# hermes-ricer` marker before injecting.

## dunst

Config: `~/.config/dunst/`
Files written: `hermes-dunstrc` fragment, injected `include` in `dunstrc`

### How it works

1. Writes `~/.config/dunst/hermes-dunstrc` with color overrides
2. Injects `include = ~/.config/dunst/hermes-dunstrc  # hermes-ricer` into `dunstrc`

### Fragment structure

```ini
# Generated by hermes-ricer — <theme-name>

[global]
    frame_color = "<primary>"
    separator_color = "<surface>"

[urgency_low]
    background = "<background>"
    foreground = "<foreground>"
    frame_color = "<surface>"

[urgency_normal]
    background = "<surface>"
    foreground = "<foreground>"
    frame_color = "<primary>"

[urgency_critical]
    background = "<danger>"
    foreground = "<foreground>"
    frame_color = "<danger>"
```

### Live reload

    pkill -HUP dunst

ricer.py should send this after writing.

### Pitfall: dunst include directives require dunst >= 1.7.0

Older versions: fragment is written but not auto-loaded. User must manually add the
include line. Check version with `dunst --version` before relying on include injection.

## Extending to New Apps

1. Create `~/.hermes/skills/creative/hermes-ricer/templates/<app>/` directory
2. Add a `.template` file with `{{palette_key}}` placeholders
3. Add a `materialize_<app>()` function in `ricer.py` following the materializer contract
4. Add `"<app>": materialize_<app>` to `APP_MATERIALIZERS`
5. If the app has no standalone binary (like KDE sub-systems), register it explicitly
   in `discover_apps()` alongside the parent system check
6. If workflow/orchestration code uses semantic element names (e.g. `terminal:kitty`,
   `gtk_theme`, `window_decorations:kde`), add or update the explicit alias layer that
   maps those names to `APP_MATERIALIZERS` keys. Do not pass category names directly to
   `ricer.py --only` unless they are real materializer keys.
7. Add fail-closed routing tests for every new alias/materializer:
   - invalid `--only=<unknown>` exits non-zero and does not emit/apply a manifest
   - invalid `--app=<unknown>` exits non-zero and does not fall back to all apps
   - workflow element aliases produce the exact `--only=<materializer>` command
   - dry-runs assert the exact `changes[].app` target set, not just exit code 0

Critical pitfall: never let an unknown filter become `None` if downstream treats `None`
as “discover/apply everything.” Targeted theming commands must fail closed; otherwise a
single unsupported workflow element can theme the entire desktop stack.

Template placeholder convention: `{{background}}`, `{{foreground}}`, `{{primary}}`, etc.
Use Jinja2 if available, built-in str.format_map as fallback.
